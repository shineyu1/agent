# SellerOS x402 买方优先前端改版设计稿

**日期：** 2026-03-25

## 概要

这次改版的目标，是把产品前端从“平台能力说明页”调整为“买方入口页”。

当前产品本身已经具备一定结构，但前端表达还没有形成足够清晰的主线。首页同时在讲服务商接入、买方发现、平台机制，结果是页面虽然有内容，但缺少明确重点，看起来更像被拼起来的模块，而不是一个有方向的产品首页。

这次改版后，前端应该让用户感受到这是一个给 Agent 使用的能力入口：

- 发现可按次付费的服务
- 判断这些服务是否值得调用
- 安装对应 Skill
- 让 Agent 进入调用流程
- 查看结果、回执与记录

服务商路径仍然保留，但不再作为首页主叙事。

## 主要用户

首页的第一目标用户应当是买方，也就是：

- Agent 开发者
- 需要把工具接入 Agent 工作流的开发者
- 想使用按次付费、可追踪结果服务的用户

因此，首页文案必须站在用户视角写，不解释平台内部结构，而是明确告诉用户：

- 你可以做什么
- 你如何开始
- 你为什么值得继续看下去

## 改版目标

- 让买方在进入首页后第一眼就明白这个产品能做什么。
- 让页面气质更锐利、更像 Web3 原生能力入口，但仍然保持克制和简单。
- 用更强的字体层级、间距和版式，替代现在偏平均化的卡片堆叠。
- 把 Skill 安装和服务发现做成真正的首页主动作。
- 加入轻量的中英双语前端切换，默认中文。
- 当前阶段只做桌面端体验，不把移动端适配纳入本轮改版范围。

## 视觉方向

整体视觉方向应当是“极简但更锋利”，而不是继续沿用当前这种普通暗色 SaaS 玻璃卡片风格。

核心原则：

- 近纯黑背景
- 更大的标题层级
- 更克制的颜色使用
- 更少的卡片
- 更大的留白
- 带一点 Web3 气质，但不靠复杂发光和堆特效撑页面

这个站点应该更像一个能力入口，而不是一个普通后台系统主页。

## 语言策略

前端需要支持两种展示语言：

- 中文，默认
- 英文，通过前端切换

切换范围包括：

- 导航文字
- 标题
- 正文说明
- 按钮
- 分区标题
- 空状态文案

不包含：

- 路由级多语言
- `/zh` 与 `/en` 分路径
- 安装命令内容翻译

语言切换按钮应放在顶栏右侧，只切换前端显示文本。

## 产品叙事调整

当前产品在页面上讲了太多平台结构，也给了服务商入口过高的前排位置。

这次改版后的首页核心承诺应当统一为：

> 发现并调用可按次付费的 Agent 服务

这句话的优点是：

- 面向买方
- 容易理解
- 能直接对应实际流程
- 能承接后续的目录、详情、调用与记录页面

## 信息架构

站点层级应调整为“买方优先”：

主要页面：

- 首页
- 服务目录
- 我的记录
- 文档

次级页面：

- 服务商接入

服务商接入仍然保留，但不应继续与首页主路径竞争注意力。

## 首页

### 首页角色

首页不是完整的平台说明书，而是买方进入产品后的第一入口。

它要完成的目标是：

- 让用户知道这是什么
- 让用户知道能做什么
- 让用户知道现在下一步该怎么开始

### 首页结构

首页建议控制在五个主要区块内：

1. Hero 首屏
2. 服务能力预览
3. 调用路径说明
4. 信任与判断信息
5. 收口 CTA

### Hero 首屏

首页首屏建议使用“中心聚焦型”版式。

内容结构：

- 标题上方的小标签
- 超大主标题
- 一小段辅助说明
- 两条 Skill 安装命令
- 复制按钮
- 两个辅助入口：浏览服务、查看文档

首屏不应该再加入：

- 服务商主文案
- 运营数据总览
- 过多说明性标签
- 大量玻璃卡片

建议文案方向：

- 主标题：发现并调用可按次付费的 Agent 服务
- 副标题：安装 Skill，让 Agent 发现服务、按次支付、直接调用，并保留结果与回执记录

需要保留的命令：

- `npx skills add agent-service-layer/user-skill`
- `npx skills add okx/onchainos-skills`

### 服务能力预览

这一屏要回答用户一个问题：

> 这里到底能给我的 Agent 用什么能力？

建议只展示 3 到 4 个能力块，不要更多。

推荐能力方向：

- Web Search
- Onchain Analysis
- Risk Scoring
- Structured Service Outputs

每个能力块只需要：

- 能力名称
- 一句用途说明
- 一个轻量标签

### 调用路径说明

这一屏用来讲清产品逻辑，不讲营销废话。

建议固定为四步：

1. 发现服务
2. 按次支付
3. Agent 调用
4. 获得结果与回执

用户需要在这一屏快速明白这个产品的运行方式。

### 信任与判断信息

这一屏要让用户在调用前，能快速判断一个服务值不值得接入。

重点展示：

- 单次价格
- 成功率
- 平均延迟
- 最近调用量
- 回执可追踪性

这里不应该做成大而杂的 dashboard，而应该做成简洁、明确、可比较的信息结构。

### 收口 CTA

首页末尾的收口应保持简洁。

主要动作：

- 浏览服务
- 查看文档

服务商接入可以作为更弱的第三入口出现。

## 顶栏导航

顶栏建议简化为以下主结构：

- 首页
- 服务目录
- 我的记录
- 文档

右侧工具项：

- 中文 / EN 切换
- 会话状态（如果当前系统有）

服务商接入不应继续作为首页级强入口。

## 服务目录页

### 页面角色

服务目录页是买方做选择和判断的地方，不是原始数据列表页。

### 改造方向

- 页头文案更明确地服务买方
- 筛选和排序更简洁
- 服务卡片更强调可比较性
- 先展示最影响决策的信息

服务卡片建议优先级：

- 服务名称
- 一句话用途说明
- 价格
- 成功率
- 平均延迟
- 最近调用量
- 详情入口

当前目录页的表达更像后台指标卡，改版后应更像“挑服务”的页面。

## 我的记录页

### 页面角色

这个页面应当代表买方自己的调用历史和回执记录，而不是平台通用仪表盘。

### 改造方向

页面应优先展示：

- 总支出
- 总调用次数
- 成功率
- 平均延迟
- 最近调用
- 失败记录及原因
- 回执入口

这个页面需要回答用户：

- 我为哪些服务付过费
- 我调用了什么
- 结果怎么样
- 证明在哪里

这比当前偏平台看板式的表达更符合买方心智。

## 服务商接入页

服务商接入仍然重要，但不再作为首页主线。

具体含义：

- 不再放在首页 Hero 中强调
- 仍然保留入口
- 定位成独立的服务商路径
- 不再与买方主转化链路竞争

## 字体与版式原则

- 标题尺寸要明显变大
- 标题字距和行距需要更精细
- 减少重复的大圆角玻璃卡片
- 页面区块之间要有更明显呼吸感
- 用更少但更有意义的容器

这次改版应主要依靠版式、文字和间距建立气质，而不是依赖特效。

## 文案原则

- 文案写给用户，不是写给项目方
- 先写用户能得到什么，再写平台怎么运作
- 减少前排位置的内部术语
- 标题更短、更明确
- 正文说明尽量压缩

产品可以复杂，但页面表达不能复杂得让用户先去理解平台架构。

## 实施前必须处理的问题

当前前端源码中存在明显中文乱码问题。这个问题必须纳入本次改版范围，因为它会直接影响：

- 文案质量
- 中英文切换一致性
- 页面可信度
- 后续维护和改文案的安全性

因此，乱码清理属于这次前端改版的必要部分。

## 本次不包含的内容

- 后端逻辑调整
- 计费机制变化
- 路由级国际化
- 服务商流程的大范围重设计
- 移动端适配与响应式优化

## 成功标准

如果改版成功，应达到以下结果：

- 买方进入首页后能立刻知道产品是做什么的
- 首页气质比当前版本更明确、更克制、更有记忆点
- Skill 安装命令成为首屏真正的主动作
- 服务目录页更像选择服务的地方，而不是指标列表
- 我的记录页更像买方的调用历史，而不是平台总览
- 中英前端切换工作稳定，默认中文
